# SysML v2 / TWC 2026x Service Playbook

Snapshot date: 2026-07-09

## Purpose

Use this when the AI needs to reason about SysML v2 service-oriented workflows in Teamwork Cloud / Magic Collaboration Studio / MagicLab-era deployments.

This file does not replace the formal OMG SysML v2 specification or 3DS product docs. It tells the agent where SysML v2 differs from classic SysML v1/Cameo plugin assumptions.

## Current standard baseline

As of the current public standards trail:

- SysML v1.7 is the latest SysML v1 minor revision, published June 2024.

- SysML v2.0 formal specifications were published in September 2025.

- SysML v2 is paired with KerML and a Systems Modeling API and Services specification.

- The Systems-Modeling SysML v2 release repository publishes example models, libraries, pilot implementation installers, API cookbook/release links, and latest incremental releases.

## 2026x Refresh1 product baseline

3DS/No Magic version news for 2026x Refresh1 reports:

- Cameo / Magic Cyber Systems Engineer / Cameo Systems Modeler 2026x Refresh1 was released June 26, 2026.

- It improves SysML v1 modeling, simulation, and client-side collaboration.

- It supports SysML v2 project migration to 2026x Refresh1.

- Teamwork Cloud / Magic Collaboration Studio 2026x Refresh1 introduces MagicLab Collaborator and SysML v2-focused API extensions.

- Those API extensions include textual API services for retrieving/updating textual representation from a namespace and graphical API services for retrieving SVG diagram visual representation by View ID.

## What changes for AI coding

### Old pattern: desktop-first SysML v1

Common route:

1. open Cameo project

2. traverse UML/profile objects

3. use stereotypes/tags

4. generate reports or mutate model via Open Java API

### Newer SysML v2 pattern: service-aware

Possible route:

1. identify SysML v2 resource / namespace / branch

2. use TWC/Magic Collaboration Studio services

3. retrieve textual representation

4. update namespace text when supported

5. retrieve graphical representation as SVG when supported

6. validate with API response and repository state

## Task routing

| Task | Use |

|---|---|

| Edit SysML v1 block/port/connector profile-based model | Cameo desktop Open Java API |

| Export SysML v1 report/table | Report Wizard / Cameo table/export |

| Inventory TWC resources | TWC REST |

| Linked model element integration | OSLC |

| Retrieve SysML v2 textual namespace | 2026x R1 textual API if enabled/licensed |

| Update SysML v2 namespace text | 2026x R1 textual API if enabled/licensed and permissions allow |

| Retrieve SysML v2 diagram SVG | 2026x R1 graphical API by View ID if enabled/licensed |

| Migrate SysML v1 concepts to v2 | Use formal transformation guidance and tool-specific migration docs; do not string-map |

## Agent questions before coding SysML v2 automation

Ask/resolve:

1. Is the target model SysML v1 or SysML v2?

2. Is it local, TWC-hosted, or MagicLab/Magic Collaboration Studio resource?

3. Which exact release line: 2024xR3, 2026x, 2026xR1?

4. Does the deployment expose the SysML v2 textual/graphical API services?

5. Is the task read-only, update, migration, validation, or report?

6. What identifiers are available: project/resource ID, branch, namespace, View ID?

7. What authentication is allowed: token, SAML, PAT, service user?

8. Are licenses/features enabled for the desired API?

## Minimal textual API workflow

```text

1. authenticate

2. resolve project/resource/branch/namespace

3. GET textual representation

4. parse or store textual model

5. validate local transformation / diff

6. PUT/PATCH update only if explicitly requested

7. retrieve again

8. compare before/after

9. log request IDs, status codes, user, timestamp

```

## Minimal graphical API workflow

```text

1. authenticate

2. resolve resource/branch/view ID

3. request graphical representation

4. save SVG as artifact

5. attach metadata: resource, branch, view ID, timestamp, API version

6. never infer model semantics from SVG alone

```

## SysML v2 migration warning

Do not claim SysML v1 and SysML v2 are interchangeable.

SysML v1 is UML-profile based in classic tooling. SysML v2 includes a language/metamodel/API-services stack and formal migration/transformation concerns.

AI-generated migration must produce:

- source construct

- target construct

- mapping rationale

- unsupported/ambiguous cases

- traceability ID

- validation rule

- human review marker

## Production logging

Every SysML v2 API automation should log:

- server base URL

- release line

- authenticated principal/service account

- endpoint path

- request method

- resource/project ID

- branch/configuration

- namespace or View ID

- status code

- response schema version if available

- output artifact hash

- before/after diff for updates
