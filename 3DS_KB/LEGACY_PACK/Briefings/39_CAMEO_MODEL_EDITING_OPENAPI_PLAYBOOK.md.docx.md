# Cameo Model Editing OpenAPI Playbook

Snapshot date: 2026-07-09

## Purpose

This is the practical model-editing guide for AI-generated Cameo / MagicDraw / CATIA Magic plugin code.

It is intentionally conservative. Editing a model is not the same as reading a model.

## Core mental model

A Cameo/MagicDraw model edit normally requires:

1. a loaded Project

2. a known scope/owner

3. a public OpenAPI path

4. a modeling session

5. element creation or mutation

6. session close/commit

7. validation and save behavior handled intentionally

## Official API rail

The vendor developer guide describes the Open Java API and separates API scope into OpenAPI, InternalAPI, and NotAPI. Code meant to extend the modeling tool should use OpenAPI, not InternalAPI or NotAPI.

The guide also states that JavaDocs are available online and in the modeling tool installation directory under openapi/docs, and that core jars are under lib while plugin libraries live under plugins/<plugin_name>.

## Minimum safe write skeleton

```java

Project project = Application.getInstance().getProject();

SessionManager sm = SessionManager.getInstance();

boolean created = false;

try {

if (!sm.isSessionCreated(project)) {

sm.createSession(project, "My safe model edit");

created = true;

}

ElementsFactory factory = project.getElementsFactory();

com.nomagic.uml2.ext.magicdraw.classes.mdkernel.Package pkg =

factory.createPackageInstance();

pkg.setName("Generated Package");

pkg.setOwner(project.getPrimaryModel());

if (created) {

sm.closeSession(project);

created = false;

}

} catch (Exception ex) {

if (created && sm.isSessionCreated(project)) {

sm.cancelSession(project);

}

throw ex;

}

```

## Editing rules

### Rule 1: Always know the owner

Do not create floating elements. Resolve the package/model/owner first.

### Rule 2: Session before mutation

Any UML/model mutation needs session handling. Never mutate elements outside a managed session.

### Rule 3: One useful command per session

Use clear session names so undo/redo and command history are useful.

### Rule 4: Read-only dry run first

Before creating or modifying elements:

- enumerate target elements

- validate target profiles/stereotypes

- preview intended changes

- write a dry-run report

### Rule 5: Never global-match stereotypes by name when profile identity matters

Preferred stereotype lookup:

1. resolve Profile by URI

2. resolve Stereotype inside that Profile

3. pass the Stereotype object to helper APIs

4. avoid name-only helper methods when multiple profiles may define same stereotype name

## Common model-edit operations

### Create a package

1. Project project = Application.getInstance().getProject()

2. ElementsFactory factory = project.getElementsFactory()

3. create session

4. Package pkg = factory.createPackageInstance()

5. pkg.setName(...)

6. pkg.setOwner(project.getPrimaryModel() or target package)

7. close session

8. validate owner contains package

### Create an element

1. choose exact metaclass

2. use ElementsFactory.create<Type>Instance()

3. set name

4. set owner

5. set required relationships

6. apply stereotypes only after profile/stereotype resolved

7. validate

### Apply stereotype

1. locate Profile by URI

2. locate Stereotype by name within Profile

3. verify canApplyStereotype(element, stereotype)

4. apply stereotype

5. set tagged values using helper APIs

6. validate values were set

### Set tagged value

1. prefer Property object when available

2. remember many tagged value getters return List even for single-valued properties

3. write values only inside a session

4. read back and normalize output

### Delete element

Deletion is high risk.

Before deletion:

- verify element type

- verify owner

- list references/dependencies

- produce dry-run report

- require explicit confirmation in operator workflow

Do not let an AI delete model elements from a vague instruction.

## Diagram vs model distinction

An element can exist in the model without appearing on a diagram.

Do not confuse:

- model element creation

- symbol creation

- diagram layout

- view refresh

If the task is "show this on a diagram", the plugin needs diagram/view APIs, not just UML element creation.

## SysML v1 vs SysML v2 caution

SysML v1 in Cameo is largely UML-profile based. SysML v2 has a distinct language/metamodel/service ecosystem. Do not translate v1 stereotypes into v2 constructs by string replacement.

## Code generation requirements for AI

Every generated plugin answer must include:

- imports with target version caveat

- plugin class or action class context

- session handling

- dry-run/validation path

- exception handling

- where to install/test

- how to verify result in the model

- what JavaDocs/classes must be checked in the local install

## Minimal operator test model

Keep a tiny .mdzip test model with:

- one root package

- one SysML profile applied

- one block

- one part property

- one port

- one connector

- one requirement

- one diagram

All mutating recipes should pass on this model first.

## Anti-patterns

- editing without a session

- catching exceptions and closing a session as if success occurred

- using user-visible names as stable IDs

- editing TWC projects without check-lock/version behavior understood

- using InternalAPI/NotAPI for production

- assuming 2024x code loads in 2026x without API review

- treating Report Wizard templates as full plugin replacements
