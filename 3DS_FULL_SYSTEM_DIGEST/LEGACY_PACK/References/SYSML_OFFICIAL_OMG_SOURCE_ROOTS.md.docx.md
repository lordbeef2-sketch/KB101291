# SysML Official OMG Source Roots

Snapshot date: 2026-07-09

## Purpose

This file pins the official OMG root pages for SysML v1 and SysML v2 so an AI does not blend the two modeling stacks.

Use this file before answering questions about:

- SysML v1 concepts, diagram types, stereotypes, and UML-profile-based modeling

- SysML v2 textual syntax, KerML, API/services, or migration strategy

- SysML v1 to SysML v2 transition/migration

- Cameo/MagicDraw SysML plugin code where the version of SysML matters

## Official roots

```text

SysML v1 root: https://www.omg.org/sysml/sysmlv1/

SysML v2 root: https://www.omg.org/sysml/sysmlv2/

```

## SysML v1 source note

OMG describes SysML v1 as a general-purpose graphical modeling language for specifying, analyzing, designing, and verifying complex systems involving hardware, software, information, personnel, procedures, and facilities.

Important v1 model-shape anchors:

- subset/extension relationship to UML 2

- XMI-based exchange expectations

- graphical diagrams for requirements, behavior, structure, and parametrics

- blocks as the basic structural unit

- block definition diagrams and internal block diagrams for structure

- requirements, satisfy, verify, parametric, and allocation concepts

AI routing:

- For classic Cameo SysML work, assume SysML v1-style UML/profile modeling unless the user explicitly says SysML v2.

- For plugins that inspect blocks, ports, connectors, requirements, stereotypes, and tagged values, load Cameo OpenAPI docs plus SysML v1 source roots.

## SysML v2 source note

OMG describes SysML v2 as the next-generation systems modeling language with enhancements over SysML v1 in precision, expressiveness, usability, interoperability, and extensibility.

Important v2 model-shape anchors:

- based on the KerML metamodel

- grounded in formal semantics

- includes textual syntax in addition to graphical syntax

- includes a standard API supporting interoperability across digital engineering tools

- intended to improve automation and modern engineering workflows

AI routing:

- For textual models, namespaces, KerML, API/services, or service-side updates, load SysML v2 roots and TWC/Magic Collaboration Studio 2026x service playbooks.

- Do not transform SysML v1 stereotypes into SysML v2 constructs by string replacement.

- A migration answer must include mapping rationale, unsupported/ambiguous cases, validation rules, and human review markers.

## Do-not-mix rule

SysML v1 and SysML v2 are related but not interchangeable.

An AI must identify which stack it is using before giving code or model-edit instructions:

```text

SysML stack: v1 / v2 / mixed migration

Tool surface: Cameo desktop OpenAPI / Report Wizard / TWC REST / OSLC / SysML v2 API/services

Model location: local .mdzip / TWC project / Magic Collaboration Studio resource

Write mode: read-only / dry-run / mutating

Source checked: OMG v1 root / OMG v2 root / vendor docs / local JavaDocs

```
