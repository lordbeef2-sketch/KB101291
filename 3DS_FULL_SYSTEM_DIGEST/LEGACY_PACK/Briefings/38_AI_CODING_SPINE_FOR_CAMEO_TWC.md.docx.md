# AI Coding Spine for Cameo / Teamwork Cloud

Snapshot date: 2026-07-09

## Purpose

This file is the missing spine for an AI that needs to code against the Dassault / No Magic / CATIA Magic ecosystem without confusing every API surface.

Use it when the task sounds like:

- write a Cameo plugin

- edit a model

- read a Teamwork Cloud project

- generate an ICD/pinout table

- use SysML v1 or SysML v2

- automate TWC

- build code for 2022x / 2024x / 2026x

This is not a mirror of proprietary vendor documentation. It is an AI routing and safety layer that points the agent to the correct surface and tells it what not to guess.

## Non-negotiable rule

Do not treat Cameo, Teamwork Cloud, SysML v1, and SysML v2 as one coding surface. They are related, but they answer different automation questions.

## Surface selection table

| User asks for | Preferred surface | Why |

|---|---|---|

| Read/export tables, docs, interface reports | Report Wizard / exports first | Lowest deployment risk |

| Custom menu action, local model traversal, validation, model edits | Cameo/MagicDraw Open Java API plugin | Runs inside the modeling client with project/session context |

| Bulk server inventory, users, projects, roles, resources | Teamwork Cloud REST | Repository/admin side |

| Linked-data / architecture management integrations | OSLC | Tool-to-tool model access semantics |

| SysML v2 textual namespace retrieval/update | TWC 2026x Refresh1 textual API if licensed/enabled | Service-oriented SysML v2 workflow |

| SysML v2 diagram visual retrieval | TWC 2026x Refresh1 graphical API if licensed/enabled | SVG/diagram representation from View ID |

| One-time data shaping after export | Python / PowerShell | Avoids plugin deployment |

| Unsupported direct .mdzip parsing | Last resort, read-only | Internal formats can change |

## Version discipline

Every generated code answer must state:

- target product: Cameo Systems Modeler / Magic Cyber Systems Engineer / MagicDraw / TWC / Magic Collaboration Studio

- target line: 2022x, 2024x, 2026x, and refresh level if known

- execution context: desktop plugin, report template, command line, TWC REST, OSLC, simulation REST, SysML v2 textual/graphical API

- Java/runtime assumption

- required plugins/licenses

- whether it is read-only or mutating

## Read-only before write

For any new automation, first build one of: inventory exporter, validator, dry-run report, or JSON/CSV preview. Only after the output is validated should the agent write to the model.

## Model editing decision tree

1. Is the model open in Cameo/MagicDraw?

- Yes: use Open Java API.

- No: can the task be done by TWC REST/OSLC/export?

2. Is the change model-content mutation?

- Yes: use SessionManager in a desktop/client plugin context.

3. Is the change diagram layout/view only?

- Use diagram APIs only after verifying target diagram/view IDs.

4. Is this SysML v2 textual content?

- Prefer TWC 2026x SysML v2 textual API where available.

5. Is the requested edit unsupported by public API?

- Refuse to guess internals; ask for exported supported artifacts or local install JavaDocs/examples.

## Red flags for AI hallucination

Stop and verify if the AI tries to:

- invent endpoint paths not present in swagger or official docs

- mutate model elements without creating/closing a session

- use stereotype names globally instead of profile URI + Stereotype object

- assume SysML v2 works like SysML v1 stereotypes

- parse .mdzip as a stable public API

- use InternalAPI / NotAPI code as if it were supported OpenAPI

- ignore refresh level compatibility

- assume all TWC deployments expose the same APIs

- call user/project/resource IDs by UI name without resolving real identifiers

## What "jdocs everything" means in this pack

The pack should not copy proprietary JavaDocs wholesale. It should store the location of local JavaDocs in each licensed tool installation, store official public JavaDoc URLs where available, index package/class names used by recipes, map every recipe to source docs and version line, and require local JavaDocs as final source before producing production plugin code.
